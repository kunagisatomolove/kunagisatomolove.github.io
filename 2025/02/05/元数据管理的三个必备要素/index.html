元数据管理的三个必备要素

元数据管理的三个必备要素

可持续元数据管理的必备工具：Metadata 360

Metadata 360是数据堆栈的整体视图，这意味着开发人员能够看到各个系统和业务中发生的情况，了解数据的生成、转换、存储和使用方式。

这在实际应用中是什么样子的？

将技术和逻辑元数据拼接在一起，无论它是数据集、ML 模型还是任务。

传统数据目录的另一个巨大问题是对仓库的不成比例的关注。我们知道，数据集不会在数据仓库中生存和消亡，它们从您的生产系统开始，通过您的流数据运行，到达仓库，然后进入您的第三方系统。将所有注意力都集中在数据仓库上，您根本无法了解全貌。

DataHub 如何实现 Metadata 360

简而言之，DataHub 是一个工具，可让您绘制和了解整个数据平台——无论您使用什么工具，也无论数据旅程处于哪个阶段。它提供集中和统一的搜索体验，以浏览您的整个数据堆栈，并根据需要对其进行切片和切块。从 ML 特征表到 Airflow 任务再到 Looker 仪表板，它可以将所有这些信息集中在一个地方。

此外，DataHub 还确保业务元数据（传统上使用术语表术语和分类法表示）连接到数据堆栈中的工具发出的物理元数据（表和列以及作元数据）。

搜索整个数据堆栈

DataHub 使跨系统和平台搜索数据实体变得容易，为您提供查找相关资源的一站式服务。

查看运营详细信息并了解它们与业务定义的关系

DataHub 将数据集或数据实体无缝映射到业务。例如，当您查看术语表术语时，您不仅会看到与之关联的代码，还会看到指标的简明语言定义。

这意味着您不必成为 SQL 专家即可正确解释它，这可确保您为更广泛的受众提供背景信息，而不仅仅是数据人员。

了解数据的端到端旅程

查看数据的端到端，以便您了解数据是如何生成的、如何转换的，以及数据如何通过各种接触点。

事实上，Airflow 管道和任务等元素显示在端到端世系视图中。

查看验证结果

借助 Great Expectations 和其他数据验证插件，您可以查看验证结果、了解发生了什么以及实际结果是什么。这有助于增强数据的信任或可用性。

Metadata 360对业务和技术团队意味着什么

让我们回到 Long Tail Companions，看看 DataHub 如何帮助不同的用户组。假设您是：

  • 寻找权威宠物数据集的数据分析工程师：

    您需要做的就是进入统一搜索体验并输入您要查找的内容。您可以按类型、数据集、图表或控制面板进行筛选，甚至可以按域进行搜索，以直接进入列表架构定义。
    在这里，您可以查看一段时间内的最近查询、热门查询、所有者和使用情况统计信息，以了解组织内部如何使用这些信息。

  • 希望了解如何计算
    宠物退货率的业务经理只需在 DataHub 中搜索 Return Rate 即可获得它的通俗易懂的定义，并了解谁拥有它，以便您可以寻求进一步的帮助。

左移

原则上，左移是指在源（即数据生成的地方）声明和发出元数据的做法。这意味着，我们不是将元数据视为事后才考虑并稍后进行注释，而是在管理和维护代码的地方发布元数据。

这很重要，原因有两个：

  1. 它帮助我们在开发人员或团队所在的地方与他们会面，而不是为了文档而将新的流程或工作流程强加给他们。
  2. 它在理解任何变化的下游影响或识别破坏性变化方面发挥着重要作用。

左移：代码中的元数据

LTC 团队可以在其 dbt 模型的架构 YAML 中使用元块来定义源元数据，如下所示。

有了这个，LTC 团队可以定义一个完全可定制的块，以捕获生成数据的代码旁边的最关键元数据，并分配：meta

  • 资产所有权
  • 模型成熟度状态（例如，生产或开发）
  • 个人身份信息状态
  • 域（常见于采用 Data Mesh 的组织中）

这样，dbt 模型的所有者就可以专注于构建模型，将其分配到不同的域，并为其分配标签 - 所有这些都在代码中完成。

DataHub 如何在其 UI 中显示在源处添加的元数据

我们还来看看 Shift Left 的另一个应用程序 — 这一次，与 LTC 的电子商务团队合作，使用 Kafka 和 Protobuf。

团队只需对其架构进行注释，同时将其添加到数据集（或在 Kafka 中称为主题）即可。

在上面的 Kafka Search Event 示例中，您可以看到一些标记为选项的其他注释，例如

  • 分类选项 （Classification.Sensitive）
  • 团队选项 （电子商务）
  • 具有敏感分类的 IP 地址字段

这种方法可确保 Schema 注释与 Protobuf Schema 并存，从而使业务上下文和业务元数据与其 Schema 保持一致。

在 DataHub 上，搜索 Search Event 会显示这些架构中直接映射到标记、术语或文档的单个元素。

此外，团队可以使用架构 Linter 来验证架构是否具有所需的注释，然后再通过其 CI/CD 管道将元数据构件推送到 DataHub。

我希望这些使用 dbt 和 Kafka 的示例能够解释如何根据不同团队的工具和开发模式定制左移原则，同时确保在 DataHub 中实现相同的发现/表面体验。

向左移动以进行影响分析

左移的另一个重要方面是将重点向左移至生产系统，以了解下游影响。

这就是在源头发出元数据有帮助的原因：它确保您拥有一个强大的知识图谱，其中包含相互依赖关系以及不同组件如何协同工作的可靠视图。正确的数据目录可以帮助您使用这些丰富的元数据进行影响分析。

DataHub 的 Lineage Impact Analysis 功能提供了获取所有资源的快照视图的能力，因此个人可以主动与人们联系，就重大变化进行对话。
